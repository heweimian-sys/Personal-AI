## 1. 为什么需要这套工作流

### 没有工作流会发生什么？

这是很多人用 AI 做项目的真实经历：

- 做了十几个项目，代码散落在桌面、下载文件夹、各种聊天窗口里
- 换了一个 AI 工具，之前的上下文全部丢失，从头再说
- 出了 bug 不知道改了哪里，也没法回退
- 做到一半的项目，隔了两周完全不知道做到哪了
- 想把项目分享给别人，发现根本无法交接

**这不是你的问题，是没有系统的问题。**

### 这套工作流解决什么？

| 问题 | 解决方案 |
|------|----------|
| 代码找不到 | GitHub 统一存档，两个仓库按用途分类 |
| AI 忘记上下文 | 项目文档作为唯一事实来源，换工具也能接着做 |
| 出了问题没法回退 | Git 版本控制，每次改动都有记录 |
| 不知道做到哪了 | 开发计划 + 更新记录，随时掌握进度 |
| 无法交接 | 标准化目录结构 + README，别人也能看懂 |

### 为什么要先建 GitHub 仓库？

很多人（包括这份 SOP 的作者）都是做到后面才知道 GitHub 的重要性——项目太多、代码太多，根本找不到。

**正确的顺序是：有想法 → 建仓库 → 再写代码。**

Git 是你的时光机，GitHub 是你的云端保险箱。不管用 Trae、Cursor、Codex 还是 Claude，所有代码最终都要存到这里。

## 2. 第一步：建立 GitHub 仓库

### 注册 GitHub（已有账号跳过）

1. 访问 https://github.com
2. 点击 Sign up，注册账号
3. 建议用常用邮箱，用户名用英文

### 新建仓库

1. 登录后点击右上角 `+` → `New repository`
2. 填写仓库名（建议中文描述，如 `Work-AI`、`Personal-AI`）
3. 选择 `Private`（私有）
4. **不要勾选** `Initialize with README`（让 AI 帮你初始化）
5. 点击 `Create repository`
6. 复制页面上显示的仓库地址（以 `.git` 结尾）

### 把仓库地址告诉 AI

把仓库地址给 AI，发这段话：

```
我在 GitHub 上创建了一个新的空仓库，地址是：
[粘贴你的仓库地址]

请帮我：
1. 在当前项目目录初始化 Git
2. 创建 .gitignore（包含 node_modules、.env、*.log、.DS_Store）
3. 完成首次 commit：feat: 初始化项目结构
4. 关联这个远程仓库并 push 到 main 分支
5. 过程中需要我授权的地方，请暂停告诉我
```

## 3. 散落项目整理：万能归档提示词

> **如果你已经有很多散落的项目，用这个提示词随时整理**

无论你用什么 AI 工具（Trae、Cursor、Codex、Claude），把下面这段复制进去，它会帮你把现有项目整理并上传到正确的 GitHub 仓库。

```
请帮我整理当前设备上的 AI 项目，并上传到对应的 GitHub 仓库。

我有两个仓库：
- 工作仓库（公司 AI 项目）：https://github.com/heweimian-sys/Work-AI
- 个人仓库（学习和实验）：https://github.com/heweimian-sys/Personal-AI

请按以下步骤操作：

第一步：扫描和分类
1. 列出你能找到的所有项目文件夹
2. 判断每个项目属于"工作"还是"个人"
3. 告诉我你的分类结果，等我确认

第二步：逐个整理（等我确认分类后）
对每个项目：
1. 检查是否已经有 .git（避免重复初始化）
2. 如果没有，初始化 Git
3. 创建或补充 .gitignore
4. 检查有没有 README/项目说明.md，没有则生成基础版本
5. commit 所有文件：chore: 归档现有项目
6. 关联对应的 GitHub 仓库
7. push 到 main 分支

第三步：汇报
列出所有已上传的项目清单，包括：
- 项目名称
- 上传到哪个仓库
- 仓库中的路径

注意：
- 遇到需要 GitHub 授权的步骤，请暂停告诉我
- 如果某个项目不确定放哪个仓库，先问我
- .env 文件里的密钥不要上传
```

```
├── 后端/

```

### AI上下文.md 模板

这是最重要的文件。AI 每次开始前读这个，立刻理解项目状态，不需要你重新解释。

```markdown
# AI上下文

## 项目介绍
[一句话描述这个项目做什么]

## 当前阶段
[现在开发到哪一步了]

## 技术栈
[用了哪些技术，如 Node.js、飞书 API、Bitable 等]

## 重要决策
[已经做过的重要技术或产品决策，不能随意更改的]

## 约束条件
[不能改的东西，比如必须用飞书 bot、不能用某个 API]

## 下一步目标
[接下来要做的事]

## 目前最大的困难
[当前最卡的问题——AI 会优先帮你解决这个]
```

**为什么要有这个文件？**
AI 没有记忆，每次对话都是全新开始。这个文件就是给 AI 的"项目交接文档"，让它每次都能快速进入状态，不需要你反复解释背景。

### 更新记录.md 模板

```markdown
## [日期] 更新记录

### 完成了什么
-

### 修改了哪些文件
-

### 下一步建议
-

### 今天如果重新开始，我会怎么做？
[AI 自动填写：经验教训，下次可以更好的地方]
```

**为什么要有"今天如果重新开始"这个字段？**
这是经验沉淀最有价值的部分。每次开发结束，AI 帮你总结踩过的坑和更好的做法。做三个项目之后，你会有一份真正属于自己的方法论。

### 初始化指令

## 6. 与 AI 协作：每日开发节奏

### 每天开始：项目交接指令

```
你现在是这个项目的长期开发成员，不是一次性的代码生成器。

请先不要写代码。

按顺序阅读：
① 项目说明.md
② 开发计划.md
③ 更新记录.md
④ AI上下文.md（重点看"目前最大的困难"）

读完后告诉我：
1. 你理解到的项目当前状态
2. 今天建议优先解决什么
3. 你的开发计划

等我确认后再开始。
```

**为什么每次都要发这段？**
AI 没有记忆。每次新对话都是白板。这段话让 AI 在动手之前先读懂项目，而不是凭着你的描述乱猜。

### 每天结束：收尾指令

```
今天开发完成了，请完成收尾工作：

1. 输出今日总结：
   - 完成了什么
   - 修改了哪些文件
   - 下一步建议

2. 更新 更新记录.md（追加今日内容，包含"今天如果重新开始"）

3. 更新 AI上下文.md（更新当前阶段和最大困难）

4. 更新 开发计划.md（勾选完成项，补充新待办）

5. Git 提交：
   - commit message 根据实际内容自动生成（feat/fix/docs 规范）
   - push 到 main 分支
   - 需要授权时暂停告诉我
```

### 遇到报错：万能提问模板

```
我在做 [功能描述]，遇到了问题。

【想实现的效果】
[一句话描述]

【实际报错】
[把终端里的完整报错粘贴在这里]

【相关代码】
[出问题的那段代码]

请先告诉我你判断问题在哪，等我确认后再给修改方案。
```

**为什么要把报错完整贴出来？**
报错信息就是诊断书，AI 从里面能读出你不知道的技术细节。不需要你先懂，只需要你把错误信息原封不动地给 AI 看。

### 需要新功能：I/O 提问模板

```
我需要增加一个功能：

输入：[描述输入是什么，比如"一个文件的 Buffer 数据"]
输出：[描述期望输出，比如"base64 格式的 md5 字符串"]
用途：[在哪里被调用，做什么用]

请先告诉我你的实现方案和会影响哪些文件，等我确认再写代码。
```

## 7. 制作 Agent 和 Skills

### 什么是 Agent？什么是 Skill？

- **Bot**：固定逻辑的自动化。触发词 → 执行预设动作。
- **Agent**：LLM 自主决策工具调用。AI 自己决定"我要调哪个工具"。
- **Skill**：可复用的能力模块，比如"读取飞书文档"、"写入 Bitable"。

你做的飞书 bot，准确说是：**Bot 作为通信载体 + Agent 作为决策内核**。飞书 bot 是手脚，LLM 是大脑。

### 制作一个新 Agent 的流程

**第一步：立项三问**

在写任何代码前，先回答这三个问题：

| 问题 | 示例 |
|------|------|
| 触发：用户做什么动作触发？ | 飞书群里 @机器人 + 输入问题 |
| 成功：成功的输出长什么样？ | 返回"文件名 + 三句话摘要 + 飞书链接" |
| 失败：找不到时回复什么？ | "未找到相关资料，请联系运营同学" |

**第二步：让 AI 拆架构**

```
我想做一个 [填你的 agent 功能]。
运行环境：[飞书 bot / 网页 / 其他]
用户操作：[用户怎么触发]
期望结果：[输出什么]

请帮我把这个系统拆成几个独立的层（摄取层、处理层、存储层、输出层），
每层写清楚：输入是什么格式 → 输出是什么格式。
不要写代码，只给我架构图和说明。
```

**第三步：逐层开发，逐层验证**

每次只调试一层，不要连着所有层一起改：

```
出问题时，按这个顺序定位：
① 数据进来了吗？→ 检查文件是否下载成功、日志有没有 error
② 内容读出来了吗？→ 检查 summary 字段是否为空
③ 内容找得到吗？→ 用关键词直接查数据库
④ Agent 能命中吗？→ 测试 query 工具，看 top3 返回什么
⑤ 输出格式对吗？→ 把正确结果单独喂给 LLM 测试
```

**原则：先跑通，再变好。**

## 8. AI 协作规则

### 开场白：贴给任何 AI 工具

每次开始新项目或换新工具，先发这段：

```
你现在是这个项目的长期开发成员，不是一次性的代码生成器。

工作原则：
1. 需求不明确时，先提问，再开发
2. 每次开发前告诉我你的计划，等我确认再开始
3. 如果可以解释，不要直接修改——先说明原因和影响范围
4. 优先保证可维护性，不追求速度
5. 不依赖聊天记录，以项目文档为唯一事实来源
6. 发现可优化的地方可以提出，但不要未经确认直接修改
7. 发现项目结构不符合规范，先问我，不要自动重构

每次回答流程：
① 告诉我你理解了什么
② 告诉我准备怎么做
③ 告诉我影响哪些文件
④ 等我确认
⑤ 再开始写代码
⑥ 完成后输出总结

现在请先阅读项目文档，理解当前进度后再告诉我你的建议。
```

**为什么需要这个规则？**

没有这个规则，AI 很容易一句话没说就开始改几十个文件，改完你不知道改了什么、为什么改、改对没有。这六步流程让你始终掌握控制权。

### 统一开发八步流程

每次开发都遵循这个顺序：

```
① 阅读项目说明.md
② 阅读开发计划.md
③ 阅读更新记录.md
④ 读取 AI上下文.md（看"目前最大的困难"）
⑤ 告知开发计划，等确认再开始
⑥ 完成后更新文档
⑦ Git Commit
⑧ Push GitHub
```

## 9. 如何维护和更新仓库

### 日常维护（每天结束时）

发"每天结束"收尾指令（见第6节），AI 会自动帮你：
- 更新三个核心文档
- 生成规范的 commit message
- push 到 GitHub

### Commit Message 规范

```
feat: 新增文件分片上传功能
fix: 修复 checksum 格式错误
docs: 更新项目说明和开发计划
refactor: 重构 agent 核心调度逻辑
chore: 更新依赖版本
```

格式：`类型: 中文描述`

### 版本迭代（每做完一个完整功能）

```
请帮我为这个项目打一个版本标签。

当前完成的功能：[列出这个版本的主要功能]
版本号建议：v[x.x]

请：
1. 更新 项目说明.md 的版本号
2. 在 更新记录.md 里新增一个版本记录
3. 创建 git tag：git tag v[x.x]
4. push tag：git push origin v[x.x]
```

### 这份 SOP 本身怎么更新

1. 每做完一个新项目，把踩到的新坑补充进"常见问题"
2. 每积累到 3 个新经验，升级版本号（v2.1 → v2.2）
3. 把更新同步到 GitHub 对应的 `知识库/SOP/` 目录

## 10. 常见问题

### Q：我不懂技术，怎么知道该问 AI 什么？

**方法只有一个：把报错信息原封不动地给 AI 看。**

不需要你先懂是什么错。终端里出现红色文字，全部复制，加上"我在做 xxx，出现了这个错误，帮我找原因"发给 AI。

### Q：AI 写了代码，我看不懂，怎么知道对不对？

不需要看懂代码，只需要验证结果：
1. 让 AI 告诉你"怎么测试这个功能"
2. 按它说的操作一遍，看结果是否符合预期
3. 如果不符合，把实际结果发给 AI

### Q：换了 AI 工具（从 Trae 换到 Cursor），之前的进度怎么办？

只要项目文档在，一切都在。在新工具里发"每天开始"的项目交接指令，AI 读完文档就能接着做。这就是为什么要维护 `AI上下文.md`。

### Q：.env 文件是什么？为什么不能上传？

`.env` 文件里存放 API 密钥、token 等敏感信息。一旦上传到 GitHub（即使是私有仓库），都存在泄露风险。规则是：密钥只写在 `.env` 里，`.env` 写进 `.gitignore`，永远不上传。

### Q：这份 SOP 适合什么阶段更新？

- 做完每个项目后，把新踩的坑补充进去
- 学到新的工具或方法后，更新对应章节
- 不需要一次写完美，跟着你的能力一起成长就好
