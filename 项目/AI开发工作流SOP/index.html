为什么第一步要建 GitHub 仓库
这份 SOP 的作者是做到后面才知道 GitHub 重要性的——项目太多、代码太多，根本找不到，也没法回退。
正确顺序：有想法 → 先建仓库 → 再写代码。
Git 是时光机，GitHub 是云端保险箱。不管用什么 AI 工具，所有代码最终都存这里。
两个仓库的分工
工
工作仓库（Work-AI）——公司 AI 项目，当前：资料抓取 Agent
个
个人仓库（Personal-AI）——学习、实验、竞品调研网站等
重要 两个仓库完全独立，永远不混用。工作代码不进个人仓库，个人项目不进工作仓库。
在 GitHub 新建仓库（只做一次）
  • 登录 GitHub，点右上角 + → New repository
  • 填写仓库名（Work-AI 或 Personal-AI）
  • 选择 Private（私有）
  • 不要勾选 Initialize with README——让 AI 帮你初始化
  • 点 Create repository，复制页面上的 .git 结尾地址
把地址给 AI，让它完成初始化
我在 GitHub 创建了一个新的空仓库，地址是： [粘贴你的仓库地址] 请帮我： 1. 在当前项目目录初始化 Git 2. 创建 .gitignore（包含 node_modules、.env、*.log、.DS_Store） 3. 按仓库规范创建目录结构和初始文档（中文模板） 4. 首次 commit：feat: 初始化项目结构 5. 关联远程仓库并 push 到 main 分支 6. 过程中需要授权的地方，请暂停告诉我
建库本身 AI 帮不了（涉及你的 GitHub 账号），但初始化的每一步都可以托管给 AI。你唯一需要手动做的就是在 GitHub 网页点几下建一个空仓库。
必须放进 .gitignore 的内容
node_modules/ .env *.log .DS_Store
API Key、飞书 token 等密钥只写在 .env 里，.env 写进 .gitignore，永远不上传。一旦泄露很难补救。

AI上下文.md 模板（最重要的文件）
# AI上下文 ## 项目介绍 [一句话描述这个项目做什么] ## 当前阶段 [现在开发到哪一步了] ## 技术栈 [用了哪些技术] ## 重要决策 [已经做的重要决策，不能随意更改] ## 约束条件 [不能改的东西] ## 下一步目标 [接下来要做的事] ## 目前最大的困难 [当前最卡的问题——AI 优先解决这个]
AI 没有记忆，每次对话都是全新开始。这个文件是给 AI 的"项目交接文档"，让它每次都能快速进入状态，不需要你反复解释背景。
每天开始
你现在是这个项目的长期开发成员，不是一次性的代码生成器。 请先不要写代码。 按顺序阅读： ① 项目说明.md ② 开发计划.md ③ 更新记录.md ④ AI上下文.md（重点看"目前最大的困难"） 读完后告诉我： 1. 你理解到的项目当前状态 2. 今天建议优先解决什么 3. 你的开发计划 等我确认后再开始。
AI 没有记忆。每次新对话都是白板。这段话让 AI 先读懂项目，而不是凭你的描述乱猜。换 Trae、Cursor、Claude 都一样用。
每天结束
今天开发完成了，请完成收尾工作： 1. 输出今日总结（完成了什么 / 改了哪些文件 / 下一步建议） 2. 更新 更新记录.md（追加今日内容，必须包含"今天如果重新开始，我会怎么做"） 3. 更新 AI上下文.md（更新当前阶段和最大困难） 4. 更新 开发计划.md（勾选完成项，补充新待办） 5. Git 提交并 push： - commit message 根据实际内容自动生成（feat/fix/docs 规范） - push 到 main 分支 - 需要授权时暂停告诉我
遇到报错
我在做 [功能描述]，遇到了问题。 【想实现的效果】 [一句话描述] 【实际报错】 [把终端里的完整报错粘贴在这里] 【相关代码】 [出问题的那段代码] 请先告诉我你判断问题在哪，等我确认后再给修改方案。
不需要你先懂是什么错。报错信息就是诊断书，AI 从里面能读出你不知道的技术细节。把红色文字全部复制过来就行。
需要新功能
我需要增加一个功能： 输入：[描述输入是什么] 输出：[描述期望输出] 用途：[在哪里被调用，做什么用] 请先告诉我你的实现方案和会影响哪些文件，等我确认再写代码。
概念先弄清楚
Bot
固定逻辑的自动化——触发词 → 执行预设动作。飞书机器人收发消息是 Bot 的部分。
AI
Agent = LLM 自主决策工具调用——AI 自己决定"我要调哪个工具"。你的 agent/core.js 升级成 Function Calling 之后才是 Agent。
Sk
Skill = 可复用的能力模块——比如"读取飞书文档"、"写入 Bitable"，可以被多个 Agent 调用。
Bot 是手脚，Agent 是大脑，Skill 是工具箱。三者组合才是完整系统。
新 Agent 的启动流程
第一步：立项三问（必做）
在写任何代码前，先回答这三个问题： 触发：用户做什么动作，触发这个功能？ 例：在飞书群 @机器人 + 输入问题 成功：成功的输出长什么样？ 例：返回"文件名 + 三句话摘要 + 飞书链接" 失败：找不到时回复什么？ 例："未找到相关资料，请联系运营同学"
第二步：让 AI 拆架构
我想做一个 [填你的 agent 功能]。 运行环境：[飞书 bot / 网页 / 其他] 用户操作：[用户怎么触发] 期望结果：[输出什么] 请帮我把这个系统拆成几个独立的层， 每层写清楚：输入是什么格式 → 输出是什么格式。 不要写代码，只给我架构说明。
先画架构，再写代码。这样出问题时你知道是哪一层的问题，不会连着所有层一起改，找不到错在哪里。
第三步：逐层验证
  • 数据进来了吗？→ 检查文件是否下载成功、日志有没有 error
  • 内容读出来了吗？→ 检查 summary 字段是否为空
  • 内容找得到吗？→ 用关键词直接查数据库
  • Agent 能命中吗？→ 测试 query 工具，看 top3 返回什么
  • 输出格式对吗？→ 把正确结果单独喂给 LLM 测试
每次只调试一层，不要连着所有层一起改。先跑通，再变好。
贴给任何 AI 工具的开场白
你现在是这个项目的长期开发成员，不是一次性的代码生成器。 工作原则： 1. 需求不明确时，先提问，再开发 2. 每次开发前告诉我你的计划，等我确认再开始 3. 如果可以解释，不要直接修改——先说明原因和影响范围 4. 优先保证可维护性，不追求速度 5. 不依赖聊天记录，以项目文档为唯一事实来源 6. 发现可优化的地方可以提出，但不要未经确认直接修改 7. 发现项目结构不符合规范，先问我，不要自动重构 每次回答流程： ① 告诉我你理解了什么 ② 告诉我准备怎么做 ③ 告诉我影响哪些文件 ④ 等我确认 ⑤ 再开始写代码 ⑥ 完成后输出总结 现在请先阅读项目文档，理解当前进度后再告诉我你的建议。
没有这个规则，AI 很容易一句话没说就开始改几十个文件，改完你不知道改了什么。这六步流程让你始终掌握控制权。

Commit Message 规范
feat: 新增文件分片上传功能 fix: 修复 checksum 格式错误 docs: 更新项目说明和开发计划 refactor: 重构 agent 核心调度逻辑 chore: 更新依赖版本
版本打标（每完成一个完整功能）
请帮我为这个项目打一个版本标签。 当前完成的功能：[列出这个版本的主要功能] 版本号建议：v[x.x] 请： 1. 更新 项目说明.md 的版本号 2. 在 更新记录.md 新增版本记录 3. 创建 git tag：git tag v[x.x] 4. push tag：git push origin v[x.x]
这份 SOP 怎么更新
  • 做完每个项目后，把新踩的坑补充进来
  • 积累到 3 个新经验，升级版本号（v2.1 → v2.2）
  • 更新同步到 Personal-AI 仓库的 知识库/SOP/ 目录
  • 飞书文档版同步更新
这份 SOP 本身也要版本管理。不需要一次写完美，跟着你的能力一起成长就好。
